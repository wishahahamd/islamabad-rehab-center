Abstract

Rehabilitation facilities handle critical, highly sensitive clinical data, including patient medical profiles, substance detox stages, and therapy progress notes. Managing these variables with manual logs or fragmented systems leads to security risks, communication delays, and audit inconsistencies. This project presents a Universal Web Portal & RBAC System built specifically for the Islamabad Rehab Center (IRC).

The system employs a 100% database-driven architecture, avoiding hardcoded routing logic, roles, or navigation menus. Built using Core PHP (PDO) and AdminLTE v4 (Bootstrap 5), it delivers a responsive portal mapped across three client workspaces: the Patient Portal (allowing recovery tracking), the Clinician Portal (intake and session logs), and the Super Admin Portal (RBAC mapping).

Furthermore, the system integrates a printer-optimized reporting engine that enables direct PDF exports, supporting both operational reports and patient clinical summaries.

1.4 Project Feasibility Analysis

1.4.1 Technical Feasibility

Core PHP is supported across standard web hosting plans and XAMPP setups. The model uses native PDO parameterized SQL inputs, neutralizing the risk of SQL injection. AdminLTE v4 runs on vanilla JavaScript (no jQuery), which keeps assets lightweight and compatible with modern web environments. The database configuration dynamically maps the application base directory pathing, preventing path resolution errors when deployed on localized servers.

1.4.2 Operational Feasibility

The system partitions clinical data into clean, role-specific views. Case Managers can execute patient intakes, Doctors can write therapy logs, and Patients can review their clinical summaries. This matches the daily workflows at the rehab facility. The responsive UI layout accommodates clinicians working on-site on desktop systems as well as staff accessing logs via mobile devices.

1.4.3 Economic Feasibility

Using an open-source LAMP stack (Linux, Apache, MySQL, PHP) avoids software licensing fees, making it an economically viable solution for the center. The hardware requirements are extremely minimal, allowing the facility to deploy the portal on their existing office computers without needing new hardware.

1.4.4 Schedule Feasibility

Structuring development into independent phases (database setup, core helpers, dashboard routing, and dynamic controls) enabled delivery within a structured five-week schedule. Parallel task allocations during the layouts and CRUD implementations optimized resources.

1.4.5 Specification Feasibility

Dynamic configuration tables make the codebase highly adaptable. Updating system branding, registering pages, or changing role permissions can be accomplished directly in the database without modifying PHP source code. This makes maintenance and future updates simple.

1.4.6 Motivational Feasibility

Replacing manual paper logs with an organized portal reduces administrative overhead for clinicians and provides patients with a transparent view of their recovery progress. This keeps staff engaged and helps motivate patients throughout their recovery process.

1.4.7 Legal & Ethical Feasibility

By restricting access through strict RBAC validation, the portal protects patient data privacy and keeps records confidential in compliance with medical data regulations. Every diagnostic report is marked confidential, preventing illegal distribution of sensitive healthcare files.

Chapter 2: Problem Defining

This chapter outlines the core operational and security challenges identified at the rehab center, which the web portal aims to solve.

2.1 Problem of Limited Reach and Accessibility

In manual workflows, patients and family members must physically visit the center to receive treatment summaries, progress updates, or counselor feedback. Outpatients lack a direct channel to review their progress notes, which often impacts patient motivation and retention.

2.2 Problem of Privacy Breaches

Paper clinical files and non-permission-controlled systems run the risk of exposing sensitive patient data, detox histories, and counselor notes to unauthorized personnel. Since drug rehabilitation records are highly sensitive, any data leak can cause serious compliance issues.

2.3 Problem of Authenticity and Trust

Without system logs and audit signatures, printed patient certificates and status summaries can be altered, creating trust and verification issues. Certifications for court sessions or employment audits require verifiable clinical histories.

2.4 Problem of Inefficient Filtering

In systems without dynamic menus, users are presented with unnecessary pages and controls, cluttering the UI and increasing operational errors. A counselor does not need to see system settings configurations, and a patient should never view logs belonging to others.

2.5 Problem of Unstructured Communication

Verbal summaries and handwritten assessment notes lead to inconsistent record-keeping between different clinicians and therapy teams. When patients move from Detoxification to Rehabilitation, critical diagnostic histories are often lost in transit.

2.6 Problem of Security and Safety

Healthcare databases are frequent targets of malicious inputs. Standard raw SQL interactions are vulnerable to injection attacks, risking data loss or unauthorized access. Furthermore, open inputs can allow Cross-Site Scripting (XSS) actions inside assessment textareas.

2.7 Problem of Manual Administration

Adding new pages, changing roles, or adjusting menu configurations manually in the code is time-consuming and prone to syntax errors. It requires constant developer assistance, which increases operational costs and delays.

2.8 Problem of Access Control

Weak access controls can allow users with lower clearance to access sensitive administrative modules or other patients' private clinical files. The system must restrict folders like /super_admin/ so they are only accessible by authorized users.

2.10 Problem of Geographical Barriers

Without secure remote access options, counseling staff and clinical directors are unable to audit patient intakes or log critical session details when working off-site. This leads to information lags and delays clinical decisions.

2.3 Functional Requirements

FR-1: Authentication & Session Protection

The system must enforce secure login credentials, hashing passwords using BCrypt (PASSWORD_DEFAULT). Active sessions must be verified on every page reload.

FR-2: Dynamic Menu Generation

The sidebar menu must load hierarchy options from sys_pages, verify if the user's role has permission to access each page, and filter out unauthorized items automatically.

FR-3: Contextual URL Access Check

Every loaded script must resolve its path relative to the root directory and confirm the active role possesses a mapping index in the role_access table for that page.

FR-4: Safe Deletion & Migrations

Deleting a custom role must block the action if it is marked as a system role. If it is a custom role with active users, the system must migrate the users to a suspended state before deleting the role to preserve database integrity.

Chapter 8: Conclusion & Future Work

8.1 Conclusion & Scope

The Islamabad Rehab Center Portal successfully implements a dynamic, role-restricted management system. By avoiding hardcoded menus, roles, or configurations, the portal can adapt to changing operational workflows at the facility without requiring code modifications. This modular design approach ensures that developers can easily extend the platform as the clinic grows.

8.2 Lessons Learnt

  • Structuring database models cleanly prevents reference integrity issues during role adjustments.
  • Centralizing redirects via JS fallback handles headers-already-sent situations securely.
  • Dynamic menus reduce UI clutter, helping clinical personnel focus on their respective workflows.
  • Database transaction rollbacks are critical when deleting roles to prevent orphaned records if a query fails.

8.3 Check Objectives

  • Objective 1: Secure clinical records? Yes. Protected by PDO statements and page access guards.
  • Objective 2: dynamic setup? Yes. Configs, roles, page URLs, and permissions are database-driven.
  • Objective 3: Printable reports? Yes. Integrates a browser print engine layout.

8.4 Future Work Scopes

While the current skeleton provides a solid foundation for clinical operations, several features can be added in future iterations:

  1. Telehealth Virtual Sessions: Integrating video conferencing libraries (e.g. Jitsi or WebRTC) directly inside the therapy log pages to support remote outpatient checkups.
  2. Automated SMS Reminders: Connecting Twilio SMS gateways to send automated appointment reminders to outpatients and family members.
  3. Inventory & Medicine Management: Adding medication tracking, dosage schedules, and inventory alerts to the clinical dashboard.
