1 System Overview

The Islamabad Rehab Center (IRC) Universal Skeleton System is a highly dynamic, enterprise-grade Role-Based Access Control (RBAC) portal built using Core PHP (PDO) and AdminLTE v4 (Bootstrap 5).

The core philosophy of the system is 100% dynamic rendering. It features zero hardcoded menus, page titles, or roles; everything is loaded dynamically from MySQL. The UI relies strictly on Bootstrap 5 styling tokens and utility classes, ensuring a cohesive and clean interface across all modules.

Core System Values:

  • Comprehensive Security Guards: Centralized authentication checking coupled with script-level RBAC queries prevents unauthorized access.
  • Safe Role Management: Custom role creations dynamically generate role keys. Protected system roles prevent deletion, and deleting custom roles migrates existing users to a suspended state automatically to preserve database integrity.
  • Dynamic Navigation Layouts: Submenus are generated recursively, automatically highlighting and expanding on active page URLs.

3 Database Schema

The relational schema comprises 7 tables (5 core infrastructure tables and 2 clinical tables):

Table 1: system_settings

Key/Value store for dynamic layout details.

CREATE TABLE system_settings (
    setting_key VARCHAR(50) PRIMARY KEY,
    setting_value TEXT NULL
);
Table 2: sys_roles

User roles catalog.

CREATE TABLE sys_roles (
    id INT AUTO_INCREMENT PRIMARY KEY,
    role_name VARCHAR(100) NOT NULL UNIQUE,
    role_key VARCHAR(50) NOT NULL UNIQUE,
    is_system_role TINYINT(1) DEFAULT 0
);
Table 3: sys_pages

Self-referencing table supporting infinite nested menus.

CREATE TABLE sys_pages (
    id INT AUTO_INCREMENT PRIMARY KEY,
    parent_id INT NULL,
    page_name VARCHAR(100) NOT NULL,
    page_url VARCHAR(255) NOT NULL,
    icon_class VARCHAR(100) NULL,
    sort_order INT DEFAULT 0,
    FOREIGN KEY (parent_id) REFERENCES sys_pages (id) ON DELETE SET NULL
);
Table 4: role_access

Mapping permissions connecting role keys to page IDs.

CREATE TABLE role_access (
    role_key VARCHAR(50) NOT NULL,
    page_id INT NOT NULL,
    PRIMARY KEY (role_key, page_id),
    FOREIGN KEY (role_key) REFERENCES sys_roles (role_key) ON DELETE CASCADE,
    FOREIGN KEY (page_id) REFERENCES sys_pages (id) ON DELETE CASCADE
);
Table 5: users

Core user account table.

CREATE TABLE users (
    id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    email VARCHAR(150) NOT NULL UNIQUE,
    role VARCHAR(50) NOT NULL,
    password VARCHAR(255) NOT NULL,
    identity_no VARCHAR(50) NULL,
    is_active TINYINT(1) DEFAULT 1,
    FOREIGN KEY (role) REFERENCES sys_roles (role_key)
);
Table 6: patients

Clinical records directory.

CREATE TABLE patients (
    id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    age INT NOT NULL,
    gender VARCHAR(20) NOT NULL,
    admission_date DATE NOT NULL,
    treatment_status VARCHAR(50) NOT NULL DEFAULT 'Intake',
    assigned_therapist_id INT NULL,
    medical_history TEXT NULL,
    FOREIGN KEY (assigned_therapist_id) REFERENCES users (id) ON DELETE SET NULL
);
Table 7: therapy_sessions

Therapist assessment logs connected to patients.

CREATE TABLE therapy_sessions (
    id INT AUTO_INCREMENT PRIMARY KEY,
    patient_id INT NOT NULL,
    therapist_id INT NOT NULL,
    session_date DATETIME NOT NULL,
    session_type VARCHAR(50) NOT NULL,
    notes TEXT NULL,
    FOREIGN KEY (patient_id) REFERENCES patients (id) ON DELETE CASCADE,
    FOREIGN KEY (therapist_id) REFERENCES users (id) ON DELETE CASCADE
);

4 Role-Based Access Control

The security design combines automated page restriction loops and dynamic menu builders:

Page-Level Access Guard:

When a page is requested, it includes header.php. The header resolves the relative URL path of the script, matches it against sys_pages, and executes checkPageAccess($pageId, $role_key).

  • If the page is registered in sys_pages, the user's active role must exist in role_access for that page ID; otherwise, execution is immediately halted, and a 403 Access Denied layout is rendered.
  • If the page is inside folders restricted to specific roles (like /super_admin/ directories), non-authorized roles are blocked even if they bypass page ID matching.

Menu Visibility Guard:

The sidebar menu loads pages and dynamically constructs a hierarchical tree using getVisibleMenuTree().

  • A child menu item is only rendered if the user's role is authorized.
  • A parent category is only rendered if the parent itself is authorized, or if it contains at least one child category that the user has permission to access. This prevents empty folders from showing.

5 Portal Workspaces

The application routes users dynamically to three specialized portals based on login configuration:

A. Patient Portal

Designed for recovering patients. It is completely isolated from system admin and clinical logs.

  • Displays admission statistics, treatment status, and therapist details.
  • Provides access to their personal therapy session progress logs.
  • Allows downloading or printing their personal Clinical Recovery Report.
B. Clinician & Doctor Portal

Tailored for Therapists, Doctors, Case Counselors, and Clinical Directors.

  • Provides dashboard summaries of active clinical files and logs.
  • Patient Intake: Full directory list and intake profiling form.
  • Therapy Logs: Session schedulers and clinical note trackers.
  • Enables exporting individual Patient Recovery Reports or downloading clinic status summary reports.
C. System Admin Portal

Restricted to Super Admins. Provides complete oversight of the dynamic skeleton:

  • Page Manager: Registers scripts, configures sort order, and updates checkbox access maps.
  • Role Manager: Handles role additions (auto-generating key values) and protects system roles.
  • User Manager: Configures login profiles and status toggles.

6 PDF Reports Engine

Printing and report downloading are handled via dashboards/rehab/print_report.php:

  • Printer-Optimized Layouts: Employs CSS print styles that hide screen buttons, letterhead double borders, and optimize font sizing for paper layout or PDF exports.
  • Auto-Print Trigger: Invokes the browser's built-in print dialog automatically after a 500ms delay to facilitate printing or saving as a PDF.
  • Privacy Validation: Enforces data compliance by restricting patient accounts to printing only their own personal records, while clinical staff can print any patient profile or facility summary report.

7 Deployment & Seeding

To install the project locally, follow these steps:

  1. Import SQL Schema: Create a MySQL database named irc_db and import schema.sql. 
    C:\xampp\mysql\bin\mysql.exe -u root -e "CREATE DATABASE IF NOT EXISTS irc_db; USE irc_db; SOURCE E:/irc/schema.sql;"
  2. Configure Constants: Review core/config.php. The database variables default to localhost root with empty password. Dynamic URL detection handles folder pathing automatically.
  3. Start Server: Start Apache inside XAMPP or trigger the built-in PHP server: 
    php -S localhost:8080

8 Testing & Verification

An automated integration test script test_suite.php is placed in the root directory to verify application health.

To execute verification: 

php E:/irc/test_suite.php

It tests DB queries, incorrect login password rejection, page permission authorizations for case counselors/admins, sidebar menu recursion visibility filters, and user migration updates upon deleting roles.
